- frontend-сборка после внесённых изменений.

## Найденные и закрытые проблемы

### 1. Копии ключа данных оставались в локальных переменных дольше, чем нужно

Файл: [vault.rs](/z:/Work%20Files/Programming/PasswordManager/src-tauri/src/vault.rs)

Проблема:

- `unlocked_snapshot()` возвращал копию `data_key`;
- эта копия использовалась в `list`, `create`, `update`;
- после завершения операций локальная копия ключа не очищалась явно.

Что сделано:

- логика переработана так, чтобы локальные копии ключа после использования очищались через `zeroize_key`;
- параллельно упрощена внутренняя структура функций для тестирования без Tauri-окна.

Статус: исправлено.

### 2. Ошибка про короткий мастер-пароль была не локализована

Файл: [i18n.tsx](/z:/Work%20Files/Programming/PasswordManager/src/lib/i18n.tsx)

Проблема:

- строка ошибки Rust `"master password must contain at least 12 characters"` не была сопоставлена с ключом локализации;
- при обходе frontend-валидации пользователь мог увидеть сырое английское сообщение даже в русской локали.

Что сделано:

- добавлен ключ локализации для этой ошибки;
- добавлен маппинг строки Rust-ошибки на перевод.

Статус: исправлено.

## Автоматические тесты

Основной автоматический контур добавлен в Rust-часть, потому что именно там находится критичная бизнес-логика:

- жизненный цикл vault-а;
- Argon2id и XChaCha20-Poly1305;
- SQLite;
- генератор паролей.

### Модульные тесты

Файл: [crypto.rs](/z:/Work%20Files/Programming/PasswordManager/src-tauri/src/crypto.rs)

Проверяются:

- детерминированность `derive_key` для одинаковых входных данных;
- изменение ключа при смене соли;
- корректный `encrypt/decrypt` round trip;
- отказ расшифровки с неверным ключом;
- отказ при невалидной длине nonce.

Файл: [generator.rs](/z:/Work%20Files/Programming/PasswordManager/src-tauri/src/generator.rs)

Проверяются:

- длина сгенерированного пароля;
- наличие символов из каждой включённой группы;
- исключение неоднозначных символов;
- отклонение длины вне диапазона;
- отклонение конфигурации без включённых групп;
- отклонение веса выше допустимого диапазона.

### Интеграционные сценарии

Файл: [vault.rs](/z:/Work%20Files/Programming/PasswordManager/src-tauri/src/vault.rs)

Добавлены сценарные тесты на полный жизненный цикл:

- отказ при слишком коротком мастер-пароле;
- полный путь `initialize -> create -> list -> update -> delete -> reset -> reinitialize -> unlock`;
- отказ разблокировки с неверным мастер-паролем;
- отказ CRUD-операций на заблокированном vault-е;
- проверка, что секреты не лежат в SQLite-файле в открытом виде.

## Остаточные риски

Ниже не баги релиз-блокеры, а реальные зоны, куда стоит идти следующим этапом.

### 1. Нет автоматических UI/E2E тестов

Сейчас хорошо покрыта backend-логика, но не покрыты браузерные/desktop-сценарии уровня интерфейса:

- ввод в формы;
- переключение вкладок;
- работа локализации в UI;
- адаптивность интерфейса при изменении окна;
- очистка буфера обмена в реальном webview.

Практический вывод:

- следующим шагом стоит добавить frontend test runner и E2E слой, например `Vitest` для unit-level frontend логики и `Playwright` для пользовательских сценариев.

### 2. Есть ветка в генераторе, которая сейчас практически недостижима

Файл: [generator.rs](/z:/Work%20Files/Programming/PasswordManager/src-tauri/src/generator.rs)

Проверка:

```rust
if groups.len() > options.length
```

в текущей конфигурации почти не срабатывает из-за того, что минимальная длина пароля уже ограничена `4`, а максимальное число групп тоже `4`.

Это не ломает приложение, но означает, что:

- проверка сейчас скорее защитная на будущее;
- как отдельный пользовательский кейс она почти недостижима.

### 3. Настройки приложения по-прежнему лежат в `localStorage`

Это нормальное решение для текущего этапа, но есть ограничения:

- настройки не участвуют в backend-политиках хранения;
- их сложнее централизованно мигрировать;
- часть поведения остаётся зависимой от frontend-среды.
